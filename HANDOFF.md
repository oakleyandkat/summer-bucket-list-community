# Summer Bucket List — Community Edition — Handoff

A real Next.js app (not the original single-file site). Multi-account summer bucket list with personal mode, group voting rooms, memory book scrapbook, an earn-and-spend coin economy, a spin-the-wheel mini-game, themed packs that unlock with coins, and AI-generated idea suggestions. Same chunky cream/sun/coral aesthetic as the original.

## Stack

- **Next.js 16** (App Router, TypeScript)
- **Tailwind v4** — custom theme tokens for the summer palette
- **Drizzle ORM** — schema-first
- **PGlite locally, Postgres on Suga in prod** — env-switched in `db/client.ts`
- **bcryptjs** for PIN hashing
- **Claude API** for AI suggestions — called via `fetch` directly (no SDK dep). Falls back to a mocked response when `ANTHROPIC_API_KEY` is missing.
- **Fonts:** Nunito (body) + Fraunces (display), via `next/font/google`

## What works

- **Accounts**: name + 4-digit PIN, bcrypt hashed, httpOnly session cookie. `localStorage` (`sbl_known_accounts`) tracks which accounts have logged in on this device so the picker shows them.
- **Personal mode**: 72 ideas grouped by 7 categories. Tap to check off — optimistic UI via `useOptimistic`, persisted in `personal_checks`. Progress bar in the dark ink stat card.
- **Surprise me**: random idea from unchecked pool, ranked by vibe if the user has taken the quiz.
- **Vibe quiz**: 7 BuzzFeed-style questions, each answer maps to mood tags. Stored as JSON in `accounts.quiz_answers`. Home page sorts/highlights ideas matching the user's mood profile via `lib/quiz.ts`'s `sortByVibe`.
- **Group rooms**: 6-letter share codes (excluding O/0/I/1/L so they're easy to read aloud). Members can vote on the 72 canonical ideas or add room-specific suggestions. Top 3 vote-getters appear in a 🏆 leaderboard with rank badges.
- **Memory book** (`/memories`): every checked idea can optionally get a photo + caption. Photos are compressed client-side (canvas, max 1200px, JPEG 0.85) and stored as base64 in the `memory` table — primary key `(account_id, idea_key)`, so re-uploading replaces. Unchecking an idea deletes its memory (no orphans).
- **Coins + earn**: every NEW check rolls a random 1–9 🪙 reward, stored on `personal_check.coins_awarded` and summed into `account.coins`. Capped at 3 rewarded checks per 10 minutes (anti-farm). The cap window uses SQL `NOW() - INTERVAL '10 minutes'` so it stays consistent regardless of any Node/PGlite timezone drift. A `+N 🪙` flyaway shows on the card when coins land, or `🐢 slow down — no coins` when farm-blocked.
- **Spin the wheel** (`/spin`): 10 🪙 per spin, atomic charge (`WHERE coins >= cost`). Weighted reward roll:
  - 🎉 JACKPOT +100 (5%)
  - 🌟 +25 (5%)
  - 🪙 +10 (25%)
  - 🌅 +5 (25%)
  - 🎟 Lucky pick — a random idea from your unchecked pool (15%)
  - 💫 Compliment — random affirmation (15%)
  - 🔮 AI pick — Claude-generated idea (10%)
- **Themed packs** (`/packs`): 3 packs (`cottagecore`, `beach`, `rainy`), each with 8 ideas, defined as code in `lib/packs.ts` (not DB rows — they're product content, not user data). Unlocking spends coins (default 50 🪙) and inserts a row in `account_pack`. Pack ideas have keys like `p-cottagecore-3` and follow the same check / memory / coin flow as canonical ideas, but `toggleCheckAction` validates that the account has actually unlocked the pack.
- **Ask the universe**: home-page button costs 15 🪙 and returns a one-off AI-generated idea matched to the user's vibe profile. Shows a clear "mock response — add ANTHROPIC_API_KEY in Suga" note when the key is missing.
- **Aesthetic**: cream background with radial sun/sky/coral gradients, floating blobs (sun, clouds, palm), Fraunces hero with "summer" in a tilted yellow squiggle, chunky neo-brutalist cards.

## Conventions worth preserving

- **All pages are dynamic.** `export const dynamic = "force-dynamic"` at the layout level. Don't pre-render — Next.js will try to query the DB at build time and crash if there's no Postgres.
- **PGlite must stay external.** `serverExternalPackages: ["@electric-sql/pglite"]` in `next.config.ts`. Without it the WASM bundling breaks.
- **Drizzle's `.returning()` takes no args** in this version. Use `.returning()` then pick fields from the row. (TypeScript types don't accept the field-spec overload.)
- **`db.client` exports `db` lazily** via a globalThis-pinned singleton — needed because Next.js dev re-evaluates modules per request, which double-opens PGlite and crashes the WASM handle.
- **Server actions** live in `app/_actions/`. **Client components** live in `app/_components/`. Pages stay in `app/`.
- **`"use server"` files can only export async functions.** No `export const SPIN_COST = 10` — keep constants local or move them to a separate non-`"use server"` module. Type exports are fine (they're erased at compile time).
- **Migrations** are applied on startup via `instrumentation.ts` (prod) and via `npm run db:migrate` (local). Generate with `npm run db:generate` after editing the schema.
- **Time-window queries should use SQL `NOW() - INTERVAL`**, not Node's `Date.now()`. PGlite's session timezone and Node's wall clock don't always agree, which causes JS-side date comparisons against stored timestamps to silently misfire.
- **No new external service deps without a chat first** — bcryptjs and postgres are already in `dependencies`; anything new needs to NOT be a build-time-only dep. The Claude API uses plain `fetch`, no SDK.

## Run / migrate / deploy

```bash
# local dev — PGlite stored at ./local.db/
npm run dev                    # next dev (uses port 3000 normally, harness uses 3001)
npm run db:generate            # after editing db/schema.ts
npm run db:migrate             # apply migrations to local PGlite

# deploy
git push                       # → Suga auto-redeploys; instrumentation.ts runs migrations on Postgres at startup
```

The Suga workspace launch config is named `sbl-community-dev` and lives in `/Users/oakley/projects/outfit_planner/.claude/launch.json` (with `--prefix` pointing here).

The Suga Postgres service has env vars `POSTGRES_PASSWORD` (Sensitive), `POSTGRES_USER`, `POSTGRES_DB`. The bucket-list service has one env var: `DATABASE_URL=postgres://postgres:bucketlist1234@postgres:5432/postgres`. The `postgres` hostname is Suga's private DNS for the postgres service.

**To enable real AI suggestions in prod:** add a `ANTHROPIC_API_KEY` env var (Sensitive) on the bucket-list service in the Suga dashboard. Get a key at console.anthropic.com. No code changes needed — the mock note disappears automatically and `lib/ai.ts` starts calling Claude.

## Known issues / gotchas

- **PGlite WASM occasionally crashes after long-idle dev sessions** with `RuntimeError: Aborted()`. Fix: restart the dev server. If that doesn't work, check `local.db/postmaster.pid` and `rm` it (stale lock from a previous crashed PGlite). If migrations still abort, `rm -rf local.db && npm run db:migrate` (wipes local accounts — production is unaffected).
- **External writes to local.db while the dev server is running don't propagate.** PGlite's singleton holds a connection that doesn't notice file changes from another Node process. If you need to edit data via a script (e.g. give an account coins for testing), stop the dev server first, do the write, then start the dev server.
- **`.env.local` is gitignored** so it won't be pushed. That's correct — Suga sets `DATABASE_URL` (and `ANTHROPIC_API_KEY` when added) via dashboard env vars in prod.
- **`local.db` is gitignored** — local PGlite database file should never be committed.
- **No passwords / no real auth**. Anyone who guesses or sees a username can view that planner. For a friend group this is fine; for real security she'd need to add Better-Auth or similar. Mention this only if she asks about privacy.
- **Schema migration approach**: drizzle-kit `npm run db:generate` after schema edits. Migrations are applied on prod startup by `instrumentation.ts`. So far this has been seamless — nothing fancy needed.
- **Tailwind 4** is in use. The planner UI doesn't actually use Tailwind much for layout — most styling is custom CSS in `globals.css` using CSS variables for the theme system. The new pack/spin/memory pages do use Tailwind utilities.
- **Memory photos are base64 in Postgres.** Rows can hit ~1MB each at the upper limit. Fine for personal use. If she goes viral, swap to a real CDN (Suga blob storage if it exists, or Cloudflare R2 / Vercel Blob). Schema change would be one-line: rename `photo` to `photo_url`, change consumer code.

### Other things she might ask for

- **Add more packs**: just edit `lib/packs.ts`, push, redeploy. No migration needed. Each new pack gets a slug, name, emoji, accent (a Tailwind color from the theme — try `sun`, `coral`, `sky`, `grass`), coinCost, and an array of `{ emoji, text, moods }`.
- **Edit pack ideas or pack costs**: same file, same workflow. Pack ideas use stable keys `p-{slug}-{n}`, so re-ordering an array would break existing checks/memories for that pack. Adding to the END is safe.
- **Tune the spin reward weights or AI ask cost**: `REWARDS` array in `app/_actions/spin.ts`, `ASK_COST` in `app/_actions/ask.ts`.
- **Change anti-farm rule**: `FARM_WINDOW_MINUTES` and `FARM_MAX_REWARDS` constants at the top of `app/_actions/check.ts`.
- **"Can I share my memory book?"**: build a public `/u/[name]/memories` read-only route. Easy — same query, different layout, omit the upload UI.
- **"Can I see my friend's planner?"**: similar `/u/[name]` route. Or rooms already support shared voting if she wants the more collaborative version.
